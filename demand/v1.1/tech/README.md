## 1. 版本变更说明

### 1.1 相比 v1.0 的主要变更

| 变更项 | v1.0 | v1.1 | 变更理由 |
|--------|------|------|----------|
| 后端架构 | Fastify + Zeabur | 微信云开发 | 简化架构，降低运维成本 |
| 实时通信 | WebSocket | 云数据库 Watch | 无需自建 WS 服务器 |
| 数据库 | PostgreSQL + Prisma | 云数据库 (MongoDB-like) | 与云开发一体化 |
| 游戏渲染 | 普通组件 | Canvas 2D | 提升游戏画面表现力 |
| 部署方式 | Zeabur 云平台 | 微信云开发 | 一站式部署，免域名备案 |

### 1.2 保持不变

- 前端框架：Taro + React + TypeScript
- 状态管理：Zustand
- PRD 需求文档：无变更

## 4. 关键技术决策

### 4.1 后端选择：微信云开发

**选择理由**：
- 与小程序深度集成，无需域名备案
- Serverless 架构，按调用量计费，朋友局场景基本免费
- 数据库 Watch 支持实时数据推送，满足游戏实时性需求
- 开发部署一体化，大幅降低运维成本

**对比其他方案**：

| 方案 | 优势 | 劣势 |
|------|------|------|
| 微信云开发 | 免运维、免备案、实时推送 | 绑定微信生态 |
| Fastify + Zeabur | 灵活、可扩展 | 需运维、成本高 |
| Firebase | 功能强大 | 国内访问受限 |

### 4.2 实时通信选择：数据库 Watch

**选择理由**：
- 无需自建 WebSocket 服务器
- 延迟约 100-300ms，对于回合制德州扑克完全够用
- 与云数据库无缝集成
- 支持断线自动重连

**工作原理**：
```
玩家操作 → 调用云函数 → 更新数据库 → 触发 Watch → 其他玩家收到更新
```

### 4.3 游戏渲染选择：Canvas 2D

**选择理由**：
- 原生 API，无需第三方库，稳定可靠
- Canvas type="2d" 支持同层渲染，解决弹窗层叠问题
- 性能足够，德州扑克场景不需要复杂渲染
- 包体积小，加载快

**对比其他方案**：

| 方案 | 优势 | 劣势 |
|------|------|------|
| Canvas 2D 原生 | 稳定、无适配问题、体积小 | 需手写渲染逻辑 |
| Pixi.js + Adapter | 功能丰富、API 友好 | 适配不稳定、体积大 |
| Cocos Creator | 专业游戏引擎 | 学习成本高、过重 |

### 4.4 状态管理选择：Zustand

**选择理由**：
- 轻量级（~1KB），适合小程序体积限制
- API 简洁，学习成本低
- 原生支持 TypeScript
- 无需 Provider 包裹

## 5. 通信协议

### 5.1 云函数调用

客户端通过 `wx.cloud.callFunction` 调用云函数：

```typescript
// 调用示例
const result = await wx.cloud.callFunction({
  name: 'room',
  data: {
    action: 'create',
    payload: { name: '我的房间', bigBlind: 20 }
  }
})
```

### 5.2 实时数据同步

客户端通过 `db.collection().watch()` 监听数据变化：

```typescript
// 监听游戏状态
const watcher = db.collection('games')
  .doc(gameId)
  .watch({
    onChange: (snapshot) => {
      // 收到游戏状态更新
      updateGameState(snapshot.docs[0])
    },
    onError: (err) => {
      console.error('watch error:', err)
    }
  })
```

## 6. 安全设计

### 6.1 认证机制

云开发使用微信 OpenID 作为用户标识，自动处理登录态：

```
小程序启动 → 调用云函数 → 自动获取 OpenID → 鉴权通过
```

### 6.2 数据安全

| 安全措施 | 实现方式 |
|----------|----------|
| 用户隔离 | 云数据库安全规则 |
| 敏感数据 | 云函数端处理，不暴露给客户端 |
| 操作校验 | 云函数内校验操作合法性 |
| 防作弊 | 发牌、结算等核心逻辑在云端执行 |

### 6.3 数据库安全规则

```json
{
  "users": {
    ".read": "auth.openid == doc._openid",
    ".write": "auth.openid == doc._openid"
  },
  "rooms": {
    ".read": true,
    ".write": false  // 只能通过云函数写入
  },
  "games": {
    ".read": true,
    ".write": false  // 只能通过云函数写入
  }
}
```
