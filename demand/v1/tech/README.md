## 3. 关键技术决策

### 3.1 前端框架选择：Taro

**选择理由**：
- React + TypeScript 语法，与团队技术栈一致
- 官方维护活跃，京东大规模生产验证
- 支持微信、支付宝、H5 等多端编译
- NutUI 组件库质量高，与 Taro 深度集成

**对比其他方案**：

| 方案 | 优势 | 劣势 |
|------|------|------|
| Taro | React 语法、跨平台、社区活跃 | 编译层可能有兼容问题 |
| uni-app | 生态最大、插件丰富 | Vue 语法、部分功能闭源 |
| 原生小程序 | 性能最佳、无抽象层 | 开发效率低、不支持跨端 |

### 3.2 状态管理选择：Zustand

**选择理由**：
- 轻量级（~1KB），适合小程序体积限制
- API 简洁，学习成本低
- 原生支持 TypeScript
- 无需 Provider 包裹，使用灵活

### 3.3 后端框架保留：Fastify

**现有优势**：
- 高性能（比 Express 快 2x+）
- 原生 TypeScript 支持
- 插件系统完善
- WebSocket 支持良好

### 3.4 数据库选择：PostgreSQL + Prisma

**现有优势**：
- 类型安全的 ORM
- 自动迁移管理
- Zeabur 原生支持
- 适合关系型数据（用户、房间、对局）

## 4. 通信协议

### 4.1 HTTP API

- 基础 URL：`https://api.holdem.example.com`
- 认证方式：Bearer Token (JWT)
- 数据格式：JSON
- 字符编码：UTF-8

### 4.2 WebSocket

- 连接 URL：`wss://api.holdem.example.com/ws/game/:roomId`
- 心跳间隔：30 秒
- 重连策略：指数退避（1s, 2s, 4s, 8s, 最大 30s）

## 5. 安全设计

### 5.1 认证流程

```
┌─────────────┐     wx.login()     ┌─────────────┐
│   小程序    │ ─────────────────▶ │  微信服务器  │
└─────────────┘                    └──────┬──────┘
       │                                  │
       │ code                             │ openid
       ▼                                  ▼
┌─────────────┐   POST /auth/wx    ┌─────────────┐
│   小程序    │ ─────────────────▶ │  后端服务   │
└─────────────┘                    └──────┬──────┘
       │                                  │
       │ JWT Token                        │ 验证 + 创建用户
       ▼                                  ▼
┌─────────────┐                    ┌─────────────┐
│   本地存储  │                    │   数据库    │
└─────────────┘                    └─────────────┘
```

### 5.2 数据安全

| 安全措施 | 实现方式 |
|----------|----------|
| 传输加密 | HTTPS / WSS |
| 密码存储 | bcrypt 哈希 |
| 令牌认证 | JWT (7 天过期) |
| 操作频率限制 | Fastify rate-limit 插件 |
| SQL 注入防护 | Prisma 参数化查询 |
